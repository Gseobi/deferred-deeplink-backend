# Design Notes

## 1. Overview

`deferred-deeplink-backend`는  
광고 링크 클릭 이후 앱 설치 및 최초 실행까지의 흐름을  
서버 기준으로 추적·검증하는 deferred deeplink 구조를 정리한 프로젝트입니다.

이 문서는 해당 구조를 선택한 이유와  
주요 설계 판단을 정리한 문서입니다.

---

## 2. Why Deferred Deeplink Needed a Server-driven Structure

deferred deeplink는 앱이 이미 설치된 사용자를 대상으로 하는 일반 딥링크보다  
상태 추적 범위가 더 길고 복잡합니다.

광고 클릭 시점에는 앱이 설치되어 있지 않을 수 있고,  
설치 이후 최초 실행 시점에 이전 유입 정보를 다시 연결해야 하기 때문입니다.

이 과정에서 중요한 것은 다음과 같습니다.

- 광고 클릭 시점 식별
- 설치 이전 상태 보존
- 최초 실행 시 유입 사용자 재식별
- 위변조 방지
- 운영 로그 및 접근 이력 추적

이 프로젝트는 위 흐름을 외부 SDK 의존보다  
서버 기준 상태 관리와 검증 흐름으로 정리하는 데 목적을 두었습니다.

---

## 3. Why Click ID Was Issued from a DB Function

Click ID는 광고 유입 식별의 기준값이기 때문에  
중복 없이 안정적으로 발급되어야 합니다.

본 프로젝트에서는 Click ID를 애플리케이션이 직접 생성하지 않고  
DB Function에서 발급하는 방향을 선택했습니다.

이렇게 한 이유는 다음과 같습니다.

- 동시성 환경에서의 중복 가능성 완화
- DB 중심 발급 규칙 명확화
- DBMS별 발급 패턴 예시 정리 가능

즉, Click ID 발급은 단순 문자열 생성이 아니라  
**광고 유입 식별 기준을 어디서 관리할 것인가**의 문제로 보았고,  
이 프로젝트에서는 DB Function 기반 접근을 선택했습니다.

---

## 4. Why an Encrypted Token Was Used

광고 링크 클릭 이후 설치 전 상태를 유지하려면  
사용자 유입 정보를 안전하게 전달할 수 있는 값이 필요합니다.

본 프로젝트에서는 Click ID 및 필요한 식별 정보를  
암호화된 `crypt` 토큰 형태로 구성했습니다.

이 구조를 선택한 이유는 다음과 같습니다.

- 설치 전 상태 보존
- 단순 파라미터 노출 최소화
- 최초 실행 시 서버 검증 가능
- 위변조 가능성 완화

즉, `crypt`는 단순 전달값이 아니라  
설치 전후 흐름을 이어주는 검증용 상태 토큰 역할을 합니다.

---

## 5. Why Access Sequence Was Managed Separately

광고 클릭 정보와 최초 실행 검증 흐름 사이에는  
중간 랜딩 및 접근 로그가 존재할 수 있습니다.

본 프로젝트에서는 랜딩 페이지 진입 시 `access_seq`를 생성해  
접근 이력과 최초 실행 검증 흐름을 연결하도록 구성했습니다.

이렇게 분리한 이유는 다음과 같습니다.

- 클릭 자체와 접근 이력을 구분할 수 있음
- 설치 전후 흐름을 더 세밀하게 추적 가능
- 최초 실행 검증 시 다중 조건 확인 가능
- 운영 로그 관점에서 흐름 파악 용이

즉, Click ID가 광고 유입의 기준이라면,  
`access_seq`는 실제 접근 흐름을 추적하는 보조 식별자 역할을 합니다.

---

## 6. Why app_type-based Routing Was Chosen

실제 운영 환경에서는 하나의 백엔드가  
여러 앱 또는 여러 광고 경로를 지원해야 하는 경우가 있을 수 있습니다.

이를 위해 본 프로젝트는 `/install/{app_type}` 구조를 사용해  
광고 대상 앱을 구분하도록 설계했습니다.

이 구조의 장점은 다음과 같습니다.

- 앱 종류별 분기 명확화
- 스토어 URL, 앱 스킴, 랜딩 정보 구분 가능
- 다중 앱 유입 구조를 하나의 백엔드에서 관리 가능

즉, `app_type`은 단순 PathVariable이 아니라  
다중 앱 광고 유입 처리 구조를 확장 가능하게 만드는 기준값입니다.

---

## 7. Why OS-based Access Control Was Included

deferred deeplink는 모바일 환경을 전제로 하는 흐름이므로  
데스크탑 환경에서 동일하게 처리하는 것은 적절하지 않을 수 있습니다.

본 프로젝트에서는 User-Agent 기반 OS 판별을 통해  
Windows / Mac 접근 시 invalidRequest 페이지를 반환하고,  
모바일(Android / iOS) 환경에 한해 deferred deeplink 흐름을 허용하도록 구성했습니다.

이 구조를 둔 이유는 다음과 같습니다.

- 모바일 전용 흐름 제어
- 비정상 접근 차단
- 운영 중 잘못된 유입 경로 구분 가능

즉, OS 판별은 단순 편의 기능이 아니라  
흐름을 올바른 환경에 한정하기 위한 접근 제어 역할을 합니다.

---

## 8. Why JPA and Querydsl Responsibilities Were Split

이 프로젝트에서는 저장과 조회/검증의 책임을 나누기 위해  
JPA와 Querydsl을 분리해 사용했습니다.

- 저장 로직: Spring Data JPA
- 조회 및 검증 로직: Querydsl QueryRepository

이렇게 나눈 이유는 다음과 같습니다.

- 저장 로직은 단순 CRUD 중심으로 유지
- 검증 및 조건 조회는 Querydsl 전용 계층에 집중
- 조회 로직 복잡도 증가에 대비
- 역할 분리로 가독성과 유지보수성 향상

즉, ORM 사용 자체보다  
**저장과 검증 조건을 어디에 둘 것인가**를 더 중요하게 보았고,  
그 결과 JPA + Querydsl 책임 분리를 선택했습니다.

---

## 9. Why Hybrid WebView Integration Context Was Reflected

이 프로젝트는 앱 자체를 개발하는 것이 목적은 아니지만,  
실제 협업 과정에서 하이브리드 WebView 환경을 함께 고려해야 했던 경험을 반영하고 있습니다.

당시 웹 측에서는 다음 역할을 수행했습니다.

- JSP 기반 랜딩 페이지 제작
- JavaScript / Ajax 기반 데이터 전달 로직 구성
- 앱-웹 호출 흐름 테스트 및 운영
- OS별 WebView 동작 차이에 맞춘 페이지 동작 확인

즉, 이 프로젝트는 Flutter 앱 구현 프로젝트가 아니라  
하이브리드 앱 환경에서 서버와 WebView가 어떻게 연결되는지를  
백엔드 및 웹 페이지 관점에서 정리한 포트폴리오입니다.

---

## 10. Why This Project Was Reconstructed as a Portfolio

실제 서비스 환경에서는 광고 유입, 설치 유도, 최초 실행 검증 흐름에 대해  
보안 및 운영 제약을 함께 고려해야 했습니다.

하지만 포트폴리오에서는 다음과 같은 제약이 있습니다.

- 실제 광고 플랫폼 연동 정보 비공개
- 실제 고객 식별 규칙 비공개
- 운영 환경 설정값 비공개

따라서 본 프로젝트는  
핵심 흐름인 Click ID 발급, crypt 생성, access 로그, 최초 실행 검증 구조를 남기고  
실제 운영 정보는 제거한 형태로 재구성했습니다.

즉, 이 프로젝트의 목적은  
deferred deeplink를 완전한 상용 시스템으로 재현하는 것이 아니라,  
**서버 주도형 딥링크 처리 구조와 검증 흐름을 설명하는 것**에 있습니다.

---

## 11. Summary

이 프로젝트의 핵심 설계 방향은 다음과 같습니다.

- 서버 주도형 deferred deeplink 흐름 관리
- DB Function 기반 Click ID 발급
- crypt 토큰 기반 상태 보존 및 검증
- access sequence 기반 접근 이력 추적
- app_type 기반 다중 앱 분기
- JPA / Querydsl 책임 분리
- 하이브리드 WebView 협업 맥락 반영

이를 통해 `deferred-deeplink-backend`는  
단순한 딥링크 예제가 아니라,  
광고 유입부터 최초 실행 검증까지의 상태 추적과 검증 흐름을  
운영 관점에서 정리한 포트폴리오로 구성되었습니다.
